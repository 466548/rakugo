# 寄席コンシェルジュ「前座さん」

> あなたの「好み」と「行ける日」から、観るべき寄席をエージェントが能動的に見つけて教えてくれる。
> 落語に興味はあるけれど「誰を観ればいいか・いつ行けばいいか分からない」初心者のための、寄席レコメンド AI エージェント。
>
> DevOps × AI Agent Hackathon 提出作品（主催 Findy / 協賛 Google Cloud Japan）

## 課題

番組表は出演者名の羅列で、香盤（出演順）やトリ（主任＝主役）の意味、芸風の暗黙知が初心者には読めない。
「この人いいな」と思っても、次にいつ・どこに出るかは各定席を巡回しないと分からない。
この **二重の壁**（誰を／いつ）が寄席から足を遠ざけている。

## やること（エージェントの必然性）

人手だと数時間かかる **判断の連鎖** を、毎日自動でこなす:

1. **解読** — 各定席の番組表（非構造データ）を取得し、日付・香盤・出演者・トリを構造化
2. **言語化** — 芸風という暗黙知を公開情報から推定し、出典つきでプロファイル化
3. **照合** — ウォッチ中の師匠 × 直近番組 × ユーザーの空き日 を突き合わせ
4. **判断・通知** — 「この日のこの席はあなた向き、理由は〜」と能動的に通知
5. **学習** — フィードバックを次の提案へ（記録はMVP / 学習はStretch）

> 検索では成立しない多段判断。だから自律エージェントである必然性がある。

## 主な機能 / 画面

| 画面 | 内容 |
|---|---|
| ホーム / 好み入力 | URL・キーワード・師匠名から芸風マッチで師匠を提案（理由つき）＋テーマ/演目への導線 |
| 演目から探す | あかね噺に出た噺など演目から、その噺を得意とする師匠へ繋ぐ（漫画ファンの入口） |
| テーマで探す | 芸風タグ（#人情噺 等）で師匠を絞り込み |
| 見取り図 | 現役噺家を「古典↔新作 × 本格↔爆笑」の2軸マップで一望（寄席で会える人・入門おすすめ） |
| 師匠詳細 | 芸風・初心者向き度・どこで会えるか・出典・出演予定（AI推定を明示） |
| 通知フィード | ウォッチ師匠 × 空き日の出演通知＋推薦理由（コア体験） |
| 番組レコメンド | 直近の対象定席の番組を初心者向き度でランキング＋解説 |
| ウォッチ / 空き日 | 追っかけ登録・行ける曜日/日付の設定 |
| 運用ダッシュボード | 自律ジョブの実行履歴・メトリクス・判断トレース（DevOps演出 兼 管理） |

## アーキテクチャ

```
[Cloud Scheduler] --daily--> [Cloud Run Job: ingest]
    取得 → Gemini構造化 → 照合(ウォッチ×番組×空き日) → 推薦生成 → Firestore
[ユーザー] <--> [Next.js on Cloud Run] <--> [Firestore] / [Gemini API (ADK)]
全工程の判断は Cloud Logging / Cloud Trace に構造化記録
```

**AIはハイブリッド設計**: 候補選定・事実（日付/部/トリ/寄席）は決定的なルールベースで正確に、
自然文の理由は Gemini(ADK) が生成。これで「AIが判断・執筆する体験」と「事実誤認ゼロ」を両立する。

### 技術スタック
- **エージェント**: ADK for TypeScript (`@google/adk`) + Gemini 2.5 Flash
- **フロント/API**: Next.js (App Router) on Cloud Run
- **自律バッチ**: Cloud Run Jobs + Cloud Scheduler
- **データ**: Firestore（本番） / ローカルJSON（開発・`DataStore` IFで差し替え可能）
- **CI/CD・監視**: GitHub Actions / Cloud Logging・Trace・Monitoring

## モノレポ構成

```
/app      Next.js フロントエンド（Cloud Run）
/agent    ADK エージェント + ツール + ドメインモデル + データ層（@rakugo/agent）
/jobs     インジェスト Job / 動作検証スクリプト（Cloud Run Jobs）
/evals    芸風分類・初心者判定・番組構造化の品質eval（CIで実行）
/infra    gcloud デプロイ手順 / Dockerfile
/.github  CI（typecheck → eval → build）/ 手動デプロイ
```

## ローカルで動かす

前提: Node.js 24.13+ / pnpm 10

```bash
pnpm install

# （任意）実Geminiを使う場合のみ。未設定でも決定的モックで全機能デモ可能。
cp env.example .env.local   # GEMINI_API_KEY=... を記入（AI Studio の無料キー）

pnpm dev                    # http://localhost:3000

pnpm ingest -- --reset      # 自律パイプラインを1回実行（ストアを初期化）
pnpm --filter @rakugo/jobs verify   # マッチング・番組構造化の動作検証
pnpm --filter @rakugo/evals eval    # 品質eval（CIと同じ）
```

データはリポジトリ直下 `.localdata/`（gitignore済み）に保存される。

## DevOps（このハッカソンの主戦場）

- **自律実行**: Cloud Scheduler → Cloud Run Job で番組取得〜推薦生成を毎日自動実行
- **観測可能性**: 各判断を `AgentRun.steps` に構造化記録 → 運用ダッシュボードで可視化（なぜこの席を推したか追える）
- **品質ゲート(eval in CI)**: `/evals` の芸風分類・初心者判定・構造化テストを GitHub Actions で実行し、品質劣化を検知
- **CI/CD**: `.github/workflows/ci.yml`（typecheck → eval → build）/ `deploy.yml`（手動・Cloud Run）

## データの正確性・倫理

- 師匠プロフィール・芸風は **AI（Gemini）が公開情報から生成した推定**であることを全画面で明示し、出典を併記
- 出演情報は番組発表時点のもの（代演＝出演者変更の可能性あり）
- スクレイピングは各定席の規約を尊重し、適切な間隔・キャッシュで行う
